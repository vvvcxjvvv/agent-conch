## 一、项目综述

### 1.1 定位

Agent-Conch 是一个全栈通用 AI Agent Harness。核心论点：**Agent = Model + Harness**——全部价值在 Harness 层，通过外部系统设计而非模型权重优化，让 Agent 在生产环境中稳定可控、可验证、可落地、可治理。

### 1.2 设计哲学

不追求模型能力堆叠，而追求工程确定性。把模型当作"能力源"，由 Harness 负责编排、约束、验证与审计。模型可以出错，但 Harness 必须保证可观测、可恢复、可治理。

### 1.3 核心工程原则

| 原则 | 落地承诺 |
| --- | --- |
| 约束解放 | RBAC + 预算熔断 + 敏感路径硬编码 + 沙箱隔离 + 安全审计 |
| 最小工具 | 13 核心工具 + 六级扩展阶梯 + 渐进发现 |
| 零信任 | LLM 评审 + 提交自审 + 运行时验证层 |
| 状态外置 | SQLite 优先，运行时状态全进 DB，不依赖模型记忆 |
| 验证前置 | 每步写操作后自动 lint/type/test，质量门禁卡点 |
| 熵增管理 | 渐进式上下文压缩 + Skill 自改进 + 轨迹压缩 |
| 简单优先 | 核心窄腰，能力通过 Skill/Plugin/MCP 扩展；YAML 驱动 |

### 1.4 0-1 阶段建设目标

四阶段递进，最终达成 P4 可治理生产 Harness：

| 阶段 | 级别 | 核心特征 | 状态 |
| --- | --- | --- | --- |
| P1 | Workflow Agent | 循环 + 工具 + 状态 + Local 沙箱 | 已完成 |
| P2 | Stateful Harness | 压缩 + 记忆 + Skill + Docker + 子 Agent | 已完成 |
| P3 | Auditable Harness | Trace + 验证 + 报告 + 搜索 + Web Console | 已完成 |
| P4 | Governable Production Harness | 权限审批 + 回归 + 调度 + 多 Agent + Web/Electron | 已完成 |

### 1.5 核心落地能力

- Observe-Think-Act 循环 + 四种错误恢复策略（retry/requery/compact/abort）
- 13 核心工具 + MCP 动态工具 + 渐进发现 + 读写分离并行执行
- 三后端沙箱（Local/Docker/SSH）+ gVisor + 网络白名单 + 快照回滚
- 渐进式上下文压缩（三步管线成本递增）+ 四层记忆 + Prompt Caching
- 运行时验证层 + 失败沉淀回归集 + 多候选 LLM 评审
- RBAC（52 权限点、五级操作、六内置角色）+ 声明式策略引擎 + 一次性写审批 + 四维预算熔断
- OpenTelemetry + Decision Trace + 双源轨迹 + Insights
- 多入口：CLI / FastAPI+SSE / React Web 工作台 / Electron 桌面端

### 1.6 适用场景与能力边界

**适用**：本地开发自动化、受控运维自动化、代码工程任务、研究型任务、单人/小团队治理工作台。

**边界**（诚实声明）：

- 单机与轻量多实例为主，不面向跨地域分布式部署
- 真实 Docker/gVisor、Vault 凭证、SSH 远端冒烟依赖外部环境，未在本地自动化中伪造
- Coordinator 为进程内编排，非分布式队列
- Electron 签名/公证/自动更新属于发布工程，未产出

### 2.2 ETCLOVG 七层分层架构

| 层级 | 职责 | 核心模块 | 依赖方向 |
| --- | --- | --- | --- |
| **E 执行环境** | 命令执行与文件操作的隔离底座 | 三后端 + FsBridge + 路径校验 + 网络策略 + 快照 | 被 T 层调用 |
| **T 工具接口** | 外部能力统一接口 | 13 核心工具 + Registry + Policy + Search + MCP + 输出管理 | 调用 E，被 L 编排 |
| **C 上下文记忆** | 模型看到什么、记住什么、遗忘什么 | ContextEngine + 压缩管线 + Caching + Skill + 四层记忆 | 被 L 调用，读写 S |
| **L 生命周期编排** | Agent 执行循环与横切能力 | AgentLoop + Layer 链 + Hook + Coordinator + Cron + 子 Agent | 编排 T/C/V/O/G |
| **O 可观测性** | 做了什么、为什么失败、成本花在哪 | OTel + Trace + Decision Trace + 轨迹 + exit_status + Insights + SSE | 被 L 调用，写 S |
| **V 验证评估** | 把"声称完成"变成"有证据完成" | 验证层 + 评审 + 自审 + 报告分离 + 回归用例 | 被 L 调用，写 S |
| **G 治理安全** | 权限审批成本合规从提示词变成系统约束 | RBAC + 策略引擎 + 写审批 + 凭证池 + 内容安全 + 审计 + 预算 | 拦截 T/L，写 S |
| **S 状态存储** | 状态外置底座 | SQLite + FTS5 + Checkpoint + 事件流 | 被所有层读写 |

**可插拔设计逻辑**：每层以 `Layer` 抽象接入引擎，`LayerManager` 按声明顺序执行钩子。新增能力只需实现 Layer 接口并注册，不改 Agent Loop 主循环。Runtime 同样可插拔，不同 Agent 类型（编码/研究/运维）共享工具、状态、验证、治理。

**分层治理逻辑**：治理不是旁路，而是工具调用链路的一环。工具执行前依次过参数校验 → 工具策略 → RBAC/策略引擎 → 写审批 → 预算，任一拦截即中止。

## 三、核心策略落地

### 3.1 工具调度

```
工具调用请求
  │
  ▼
查找工具 ──(未找到)──► 错误记录
  │
  ▼
Pydantic 参数校验（先过滤多余字段再实例化）
  │
  ▼
工具策略三层：Allow/Deny 显式 → 规则评估 → 默认允许
  │
  ▼
RBAC + 声明式策略引擎（RBAC 先行 → 规则匹配 → 风险阈值审批）
  │
  ▼
写审批（受保护路径：请求哈希防篡改 → pending 复用 → 批准后一次性消费）
  │
  ▼
预算检查（Token/时间/工具次数/资源，超限 → BUDGET_EXCEEDED）
  │
  ▼
执行工具 → 记录健康状态（成功计数 / 失败计数）
  │
  ▼
结果脱敏 → 长输出制品化（0600 私有制品 + 预览引用）
```

**0-1 技术难点**：LLM 传入冗余参数导致校验失败。**解决方案**：参数校验前先按 input_model 字段过滤，再实例化。check_fn 每次暴露给模型前调用会很慢（可能涉及网络检查），用 30s TTL 缓存 + 连续失败 ≥2 次进入 60s 瞬态抑制。

### 3.2 并行工具执行

```
模型返回多个工具调用
  │
  ├─ 按 is_write_tool 分离
  │
  ├─ 读操作集合 ──► asyncio.gather(并行执行, return_exceptions=True)
  │                  └─ 异常不中断其他读
  │
  └─ 写操作集合 ──► 逐个串行执行
                     └─ 避免文件竞争
  │
  ▼
按原始调用 ID 排序合并结果
```

**设计取舍**：只对互不依赖的读操作并行；写操作和危险操作串行。这是可靠性优先于性能的决策。

### 3.3 渐进式上下文压缩

```
上下文超预算？
  │
  ▼
Step 1: ResultCleanup（零 LLM 调用）
  ├─ 清理 >200 chars 的旧工具结果
  ├─ 保留最近 10 条
  └─ 仍超预算？──是──► Step 2
                    └─ 否──► 完成
  │
  ▼
Step 2: ContentFolding（零 LLM 调用）
  ├─ 对 >2000 chars 内容折叠
  ├─ head 900 + tail 500，中间标记 collapsed N chars
  └─ 仍超预算？──是──► Step 3
                    └─ 否──► 完成
  │
  ▼
Step 3: SummaryArchive（一次 LLM 调用）
  ├─ 结构化摘要：Historical / In-Progress / Pending / Remaining
  ├─ 添加 REFERENCE ONLY 前缀 + end marker
  └─ 提取 Compact Attachment（recent_files / discovered_tools / async_tasks）
```

**自研差异化**：三步管线成本逐步递增，能用确定性规则解决就不调 LLM。铁律：不允许变更过去上下文、不切换 toolset、不重建 system prompt，唯一例外是压缩。

```
  │
  ▼
按 YAML 串行执行验证命令：
  lint → type-check → test
  │
  ├─ 首个失败即停止
  ├─ 保留末尾 4000 字输出
  │
  ▼
生成验证报告：
  agent_claim（Agent 自述，不可信）
  checks（服务级验证结果，可信）
  │
  ├─ 通过 ──► 放行
  │
  ▼
失败
  │
  ├─ 注入修复消息让模型重试
  ├─ 阻止进入下一步（质量门禁）
  └─ 失败用例经 SHA256 指纹去重后沉淀为回归用例
      └─ 回归运行器批量执行 → gate_passed / pass_rate 门禁
```

**自研差异化**：从"声称完成"升级为"有证据完成"；agent_claim 与 checks 互不背书；失败即资产，沉淀为回归。

### 3.7 安全策略引擎

```
操作请求
  │
  ▼
RBAC 先行：角色 → 权限点 → 操作级别（READ/WRITE/EXECUTE/ADMIN/CRITICAL）
  │
  ├─ 拒绝 ──► 终止
  │
  ▼
声明式规则匹配（受控子集：roles/senders/tools/actions/level/argument_contains）
  │
  ├─ DENY ──► 终止
  ├─ ALLOW ──► 放行
  │
  ▼
REQUIRE_APPROVAL
  │
  ├─ 生成审批请求（请求哈希防篡改）
  ├─ pending 状态可复用
  ├─ 批准后一次性消费
  └─ 恢复原始工具请求
  │
  ▼
内容安全：按字段语义扫描内联密钥（非任意字符串，减少误报）
  ├─ 工具结果与最终回答统一脱敏
  └─ 敏感内容外发阻断
  │
  ▼
预算检查（四维实时记账，超限 → BUDGET_EXCEEDED）
```

**设计取舍**：未引入 OPA/CEL，用受控声明式子集减少依赖和动态执行风险。复杂布尔表达式需后续扩展。

### 3.8 错误分类恢复

```
模型调用 / 工具执行异常
  │
  ▼
错误分类器（31 种错误类型）
  ├─ API 超时 / 限流 / 认证 / 内容策略 / 连接
  ├─ SSL 证书 / 服务器错误 / 错误请求 / 未找到 / 过载
  ├─ 工具执行 / 工具超时 / 工具阻塞 / 工具校验
  ├─ 上下文窗口超限 / 权限拒绝 / 沙箱不可用 / 沙箱超时
  ├─ 格式错误 / 成本超限 / Max Tokens / JSON 解析
  ├─ 数据库错误 / 未知
  │
  ▼
归类到恢复策略：
  ┌──────────┬──────────────────────────┐
  │ retry    │ 退避重试（超时/限流/连接）│
  │ requery  │ 重新提问（格式/内容策略） │
  │ compact  │ 压缩上下文（窗口超限）    │
  │ abort    │ 终止（认证/SSL/不可重试集）│
  └──────────┴──────────────────────────┘
```

**踩坑教训**：循环内 except 曾静默吞异常，导致 await 同步方法触发的 TypeError 被捕获，break 未执行，循环跑满 max_turns。教训：循环内 except 不静默吞异常，需记录详细错误或打印 traceback。

## 五、项目复盘总结

### 5.1 性能瓶颈

| 瓶颈 | 现状 | 影响 |
| --- | --- | --- |
| 同步 SQLite + 异步事件循环 | sqlite3 stdlib 同步 API，用 `check_same_thread=False` + `to_thread` 规避 | P1 阶段操作 <1ms 可接受；高并发写入会成为瓶颈 |
| Token 计数用 4 chars ≈ 1 token 估算 | 未用 tiktoken 精确计数 | 压缩阈值判断有偏差，可能过早或过晚压缩 |
| 单 SQLite 写者 | 所有层共用一个 DB 文件 | 高频写入（轨迹+Trace+事件）可能竞争 |
| FTS5 降级 LIKE | 部分 SQLite 编译无 FTS5 | 跨会话搜索性能下降 |

### 5.2 功能短板

| 短板 | 说明 |
| --- | --- |
| 无真实 Docker/gVisor E2E 证据 | 本地无 daemon，仅异步接口测试 + API 409 |
| 无真实 Vault 凭证 CI | Bitwarden/1Password 依赖已登录 CLI，未做真实账户验证 |
| Coordinator 单机 | 无法跨节点调度，无抢占 |
| Cron 固定 UTC | 无时区/DST 规则 |
| 自审为确定性规则 | 不发起额外 LLM，语义深度有限 |
| Skill Improve 为确定性模板 | 未用 LLM 生成改进内容 |

### 5.3 潜在风险

| 风险 | 说明 |
| --- | --- |
| 规则 DSL 表达力不足 | 受控子集无法覆盖复杂布尔逻辑，可能被迫硬编码 |
| 事件总线单进程 | 多实例部署时 SSE 事件不共享（已用 SQLite polling 部分缓解） |
| 沙箱逃逸面 | Local 后端默认信任，依赖 PathValidator 路径级保护 |
| 凭证 CLI 依赖 | bw/op 未安装或未解锁时凭证池不可用 |
| 前端无 E2E | UI 回归依赖手动验证 |

### 5.4 短期优化方案（可落地）

| 优先级 | 方案 | 预期收益 |
| --- | --- | --- |
| 高 | 切换 tiktoken 精确 Token 计数 | 压缩阈值准确，成本估算精确 |
| 高 | 接入 OTLP exporter（可配置） | 跨服务 Trace，生产可观测 |
| 高 | Web Console 引入 Playwright E2E | UI 回归自动化 |
| 中 | LongTermMemory 接入向量检索（sqlite-vss 或外部） | 长期记忆召回质量提升 |
| 中 | Coordinator 支持 Redis 队列 adapter | 跨节点调度能力 |
| 中 | aiosqlite 迁移 | 高并发写入非阻塞 |
| 低 | Cron 支持时区 | 多时区场景友好 |
| 低 | Skill Improve 接入 LLM 提案生成器 | Skill 自改进深度提升 |
