## 文档一：阶段整体实现进度总结
文件名：`Agent-Conch_{阶段标识}_阶段实现进度总结.md`

### 一、阶段总览
- 阶段定位与核心目标回顾（引用设计文档定义）
- 计划周期与实际完成周期
- 整体完成度（百分制 + 一句话结论）
- 核心交付物完成情况总表

### 二、交付物逐项核对
严格对照设计文档中对应阶段的「交付物表格」，逐项核验：
| 模块 | 设计要求 | 实现状态（已完成/部分完成/未启动） | 完成度说明 | 关联代码路径 |
| ---- | -------- | ---------------------------------- | ---------- | ------------ |

### 三、架构分层完成度总览
按 ETCLOVG 七层 + S 状态层逐层汇总：
| 层级 | 层级名称 | 本阶段计划能力 | 实际覆盖情况 | 核心差异 |
| ---- | -------- | -------------- | ------------ | -------- |

### 四、关键设计偏差说明
针对所有与原始设计不一致的实现，逐条说明：
1. 偏差点描述
2. 原始设计方案
3. 实际实现方案
4. 偏差原因（技术限制/成本优化/需求变更/架构简化等）
5. 影响范围与后续修复计划

### 五、验证结果汇总
- 设计文档中验证标准的达成情况
- 核心用例通过率
- 关键指标数据（耗时、Token 消耗、成功率等）
- 未通过验证项说明与根因

### 六、遗留问题与技术债
按优先级（高/中/低）罗列：
- 问题描述与影响范围
- 临时规避方案
- 建议修复阶段

### 七、下一阶段前置建议
- 本阶段沉淀的可复用能力
- 下一阶段的依赖条件与风险
- 架构与技术栈调整建议

## 文档二：分层技术实现沉淀
按 ETCLOVG 七层 + S 状态层，每层生成独立沉淀文档，文件名格式：`{层级标识}_{层级名称}_层实现策略沉淀.md`
（示例：`T_工具接口层实现策略沉淀.md`）

每层文档统一包含以下章节：

### 一、设计目标回顾
- 本层在整体架构中的定位与核心价值
- 本阶段计划达成的核心能力项
- 设计文档中的核心约束与工程原则

### 二、核心实现方案
- 本层整体实现结构（文字说明或 ASCII 架构图）
- 核心类/接口设计与职责说明
- 关键业务流程实现逻辑
- 核心代码文件路径索引

### 三、设计落地对照
- ✅ 完全对齐设计的能力项
- ⚠️ 调整/简化/增强的能力项
- 调整原因与合理性说明
- 对整体架构的影响评估

### 四、关键技术点与踩坑记录
- 实现过程中的核心技术难点
- 选型决策过程与最终方案
- 踩坑记录与规避建议
- 性能/稳定性优化手段

### 五、验证与覆盖情况
- 验证范围与测试用例设计
- 验证结果与核心数据
- 未覆盖的边界场景与异常情况
- 已知的能力边界

### 六、演进与优化方向
- 当前实现的局限性
- 下一阶段的优化计划
- 长期能力演进路线

# 写作约束
1. 客观写实：禁止模糊完成度，未实现能力必须明确标注，不得用「基本完成」等模糊表述
2. 可追溯：所有结论必须对应具体实现与代码路径，禁止空泛描述
3. 问题导向：重点记录偏差、踩坑与遗留问题，技术沉淀价值优先于成果展示
4. 工程视角：围绕稳定性、可维护性、可扩展性、成本四个维度评估实现质量
